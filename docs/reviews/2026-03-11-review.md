## Warnings (Should Fix)

### W1. Double horizontal padding on centered fullscreen layouts

**Files:** `src/app/report/[slug]/not-found.tsx`, `src/app/report/[slug]/page.tsx` (generating and failed states)

These pages wrap a `Container` inside a div that itself has `px-4 sm:px-6`:

```tsx
<div className="flex min-h-screen flex-col items-center justify-center px-4 sm:px-6">
  <Container variant="prose">
    ...
  </Container>
</div>
```

At mobile widths (375px), the Container's `max-w-(--container-prose)` (42rem = 672px) exceeds the viewport, so the Container's internal `px-4` stacks with the outer div's `px-4`. This produces 32px of padding on each side, leaving only 311px of content width. On narrow screens with centered text this is noticeable.

**Recommendation:** Remove `px-4 sm:px-6` from the outer wrapper div in these three locations and let Container's padding handle it alone. Or remove Container's padding for these cases and keep only the outer padding. The key is to not stack two identical responsive padding declarations.

### W2. `SectionErrorBoundary` retry may re-throw the same error indefinitely

**File:** `src/components/SectionErrorBoundary.tsx`, line 48-50

The retry handler resets `hasError` to `false`, which re-renders the children. If the error is deterministic (e.g., a null-access on data that will always be null), clicking "Try again" will immediately crash again, creating an infinite crash-retry loop from the user's perspective.

**Recommendation:** Consider adding a retry counter and, after 2-3 retries, showing a permanent fallback message like "This section could not be loaded." This prevents the user from endlessly clicking retry on deterministic errors.

## Suggestions (Consider)

### S1. Missing test for `SectionErrorBoundary`

No tests exist for the error boundary component. This is understandable given the project currently lacks a DOM testing environment (no `@testing-library/react`). However, the error boundary is a critical resilience component. When DOM testing is added in the future, verifying that `SectionErrorBoundary` catches child render errors and displays the fallback should be a priority.

### S2. `overflow-x: hidden` on `html` can mask layout bugs

**File:** `src/app/globals.css`, line 113

Setting `overflow-x: hidden` globally prevents horizontal scroll, which is the goal. However, it also masks elements that accidentally overflow (e.g., a misconfigured absolute-positioned element or an unresponsive iframe). This makes layout bugs harder to detect during development.

**Recommendation:** Consider applying `overflow-x: hidden` only on `body` instead of `html`, or using it as a development-time debug aid (temporarily removing it to catch overflow sources). This is a minor point -- the current approach is pragmatic and common.

### S3. Consider `"failed"` status in Cache-Control test coverage

**File:** `src/lib/__tests__/report-route-slug.test.ts`

Tests cover Cache-Control for `"complete"` and `"generating"` statuses. The `"failed"` status is also non-cacheable but lacks its own test. While the implementation handles it correctly (anything not `"complete"` gets `no-cache, no-store`), an explicit test for `"failed"` would strengthen the assertion.

### S4. Education grid: odd item count on 2-column mobile layout

**File:** `src/components/sections/DemographicsSection.tsx`, line 155

The Educational Attainment section uses `grid-cols-2 sm:grid-cols-3`. When all three education stats are present, the mobile layout shows 2 items in the first row and 1 orphan item in the second row. This is a minor aesthetic issue -- the orphaned card will be left-aligned and half-width.

**Recommendation:** Consider using `grid-cols-1 sm:grid-cols-3` for this specific section to avoid the orphan, or keeping it as-is if the 2-column layout is preferred for density on mobile.

### S5. Dynamic Map loading skeleton could match shadow style

**File:** `src/components/ReportContent.tsx`, lines 39-44

The Map's dynamic import loading skeleton does not include the `shadow-sm` class that the actual Map component receives via the `className` prop. This means there's a subtle visual shift when the Map loads (no shadow -> shadow). Minor, but adding `shadow-sm` to the skeleton wrapper would eliminate it.

### S6. `appearance-none` on AddressInput may affect accessibility

**File:** `src/components/AddressInput.tsx`, line 186

The `appearance-none` utility removes browser-default styling from the input. On most modern browsers this is fine, but on some older mobile browsers it can remove focus indicators or native affordances. The existing `focus:ring-2 focus:ring-accent-subtle` ensures focus visibility is preserved, so this is likely fine. Just flagging for awareness.

## Convention Compliance

The implementation follows existing CONVENTIONS.md patterns correctly:

- **Error boundary as class component:** Correctly noted that React requires class components for error boundaries.
- **`"use client"` directive:** Present on the error boundary (required since it uses state and event handlers).
- **Console error logging:** Uses `[SectionErrorBoundary]` bracketed tag for grep-ability, matching the project pattern.
- **Dynamic import pattern:** Uses `next/dynamic` with `ssr: false` and a loading skeleton, matching the new convention added in this phase.
- **Cache-Control headers:** Properly set via `NextResponse.json()` options object, consistent with the API route patterns.
- **Responsive patterns:** Consistently uses `[mobile] sm:[desktop]` pattern across all modified files.
- **Container padding:** Correctly centralized to `px-4 sm:px-6` in Container, and outer `px-6` wrappers were removed from sections that use Container (except for the double-padding issue noted in W1).
- **Test patterns:** New tests follow the existing mock structure, use `vi.fn()` / `mockResolvedValueOnce`, and import from `vitest`.
- **CONVENTIONS.md updated:** Three new sections added (Responsive Design, Performance, Error Boundaries) with clear, actionable guidance.
